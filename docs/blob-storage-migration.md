# Blob 主路径迁移可行性分析

## 背景

当前项目的图片主链路以 `dataUrl` 为中心：

- API 返回的图片最终以 `dataUrl` 字符串形式进入运行时。
- IndexedDB 的 `images` 表当前直接存 `dataUrl`。
- 内存缓存 `imageCache` 当前也是 `Map<string, string>`，缓存值为 `dataUrl`。
- 导入导出链路会在 `dataUrl` 与二进制 bytes 之间来回转换。

这套方案实现简单、兼容性好，但在高分辨率图片场景下有明显成本：

- `dataUrl` 会比原始二进制更大。
- 大图会产生更多字符串拼接、复制、编码和哈希成本。
- 对 `2K/4K` 图片来说，`dataUrl` 更容易放大内存峰值和主线程压力。

## 当前存储现状

### 1. IndexedDB

当前 `src/lib/db.ts` 中：

- `StoredImage` 以 `dataUrl` 为主字段。
- `storeImage(dataUrl)` 直接按 `dataUrl` 哈希并落库。
- `hashDataUrl(dataUrl)` 对整段字符串做哈希。

### 2. 内存缓存

当前 `src/store/cache.ts` 中：

- `imageCache` 结构为 `Map<string, string>`。
- `ensureImageCached()` 从 DB 读取后，把 `record.dataUrl` 重新放回缓存。

### 3. 运行时

当前 `src/store/runtime.ts` 中：

- 输入参考图、局部编辑蒙版、输出结果图都围绕 `dataUrl` 传递。
- 输出图追加时直接 `storeImage(dataUrl, 'generated')`。

### 4. 导入导出

当前 `src/store/dataTransfer.ts` 中：

- 导出时会把本地图片的 `dataUrl` 解成 bytes 再打包进 zip。
- 导入时会把 zip 内 bytes 重新编码回 `dataUrl` 再落库。

## 什么是 Blob 主路径

Blob 主路径的含义是：

- 图片在前端内部的“标准形态”从 `dataUrl` 字符串切换为 `Blob` / 二进制。
- DB 主存储优先存 `Blob`，而不是存 `dataUrl`。
- 读取时通过 `URL.createObjectURL(blob)` 得到 `blob:` URL 供 UI 使用。
- 哈希优先针对 `Blob` 的二进制内容，而不是针对 `dataUrl` 字符串。

## 优点

### 1. 更省内存

- `dataUrl` 包含 `data:` 前缀和 base64 膨胀。
- Blob 主路径不需要长期持有超长字符串。

### 2. 更适合大图

- `2K/4K` 图片在 Blob 形态下更稳定。
- 可以减少字符串复制、编码和扫描的开销。

### 3. 为后续优化留空间

- 更容易配合 `Worker` 做解码或后处理。
- 更容易做缩略图缓存、延迟解码和分层存储。
- 更适合后续继续优化大图导入导出。

### 4. 存储语义更自然

- 图片本质是二进制资源，Blob 表达更贴近资源本身。
- `dataUrl` 更适合作为兼容显示形式，而不是主存储形式。

## 缺点

### 1. 改造面较广

这不是单点改动，会影响：

- 类型定义
- DB 层
- 内存缓存层
- 运行时图片读取/回填
- 导入导出
- 部分 API 辅助函数

### 2. 需要兼容旧数据

现有用户本地 IndexedDB 里已经存了大量 `dataUrl`。
迁移时需要支持：

- 老数据继续可读
- 新数据优先走 Blob
- 平滑过渡而不是一次性清库

### 3. UI 读图方式要调整

现在很多地方天然把图片当字符串 `src`。
改成 Blob 后，需要在读取层生成 `blob:` URL，再把 URL 交给 UI。

### 4. 导入导出链路要重新梳理

当前导入导出天然围绕 `dataUrl` 设计，Blob 主路径后最好改成“Blob/bytes 优先，dataUrl 仅兼容”。

## 改造量评估

综合看，这属于：

**中等偏大的基础设施改造**

不是推倒重写，但也不是小修小补。

如果一次性全量切换，风险偏高；如果采用兼容迁移，风险可控。

## 推荐迁移策略

不建议一步到位改成纯 Blob 模式，建议分阶段做。

### 第一阶段：数据模型兼容

目标：

- `StoredImage` 允许同时支持 `dataUrl`、`blob`、远程 URL。
- DB 层先支持双格式读写。

建议：

- 保留旧字段兼容读取。
- 新增 Blob 字段或等价二进制存储字段。
- 新图优先按 Blob 落库。

### 第二阶段：缓存层兼容

目标：

- 缓存不再只缓存 `dataUrl`。

建议：

- 缓存层支持 `Blob` 与 `blob:` URL。
- 为 UI 提供统一的“可显示图片地址”获取函数。

### 第三阶段：运行时逐步切换

目标：

- 输出图主存储从 `dataUrl` 切换为 Blob。

建议：

- API 返回后尽快转为 Blob 或保持二进制。
- `storeImage` 新增 Blob 版本。
- 老的 `storeImage(dataUrl)` 保留一段时间作为兼容入口。

### 第四阶段：导入导出收口

目标：

- 导出时优先从 Blob/bytes 打包。
- 导入时优先直接写入 Blob。

建议：

- ZIP 格式可以保持不变。
- 只是内部不再先转回完整 `dataUrl` 再落库。

## 对当前项目的现实建议

当前项目已经通过 Responses SSE 拆分链路解决了最痛的高分辨率稳定性问题。

因此 Blob 主路径改造当前不属于阻塞项，更适合作为下一阶段优化：

- 当图库规模继续变大时
- 当 `2K/4K` 图片数量继续上升时
- 当需要进一步压缩内存占用时
- 当需要引入 `Worker` / 缩略图 / 分层缓存时

## 结论

Blob 主路径迁移：

- **可行**
- **收益明确**
- **改造量中等偏大**
- **最适合采用兼容式分阶段迁移**

对于当前项目，最佳策略不是立即实施，而是：

1. 保留现有稳定的 Responses 拆分链路。
2. 在后续需要进一步优化大图内存和存储成本时，再启动 Blob 主路径迁移。

## 后续可选优化路线

以下优化都具备可行性，但当前都不属于必须立刻实施的工作。

### 1. Blob 主路径重构

定位：

- 中等偏大的基础设施升级
- 面向长期内存占用、存储成本、大图库规模优化

适用时机：

- 图库规模继续增长
- `2K/4K` 图片成为常态
- 需要继续压低内存峰值
- 需要配合 `Worker`、缩略图、分层缓存继续优化

预期收益：

- 减少 `dataUrl` 带来的字符串膨胀
- 降低大图哈希、缓存、落库成本
- 让图片主存储更贴近二进制资源本身

主要风险：

- 改造面较广
- 要兼容旧的 `dataUrl` 数据
- 需要同步调整缓存、运行时、导入导出

建议顺序：

1. `StoredImage` 双格式兼容
2. DB 与缓存层兼容 Blob
3. 运行时逐步切换新图到 Blob 主路径
4. 导入导出最后收口

### 2. SSE parser 进一步做成更细的字节级处理

定位：

- 更底层的流式解析优化
- 面向极端大 SSE 响应、进一步降低字符串复制成本

适用时机：

- 后续再次遇到更大的单事件 SSE
- 浏览器长时间批量生成时仍出现明显内存增长
- 需要为更极端的流量规模预留稳态余量

预期收益：

- 更早在字节流阶段识别事件边界
- 降低字符串拼接、切片和中间对象生成
- 理论上更接近“超大 SSE 也稳”的方向

主要风险：

- 实现复杂度明显上升
- 调试难度更高
- 容易引入边界问题，例如：
  - 跨 chunk 边界
  - 多字节字符
  - SSE 空行与 `data:` 拼接规则

当前建议：

- 暂不实施
- 当前项目在现有拆分链路下已经稳定支撑 `Responses 2K/4K`
- 只有在后续再次碰到极限场景时，才值得投入

### 3. 继续压榨字符串扫描细节

定位：

- 中低风险的小步优化
- 在不重写底层架构的前提下继续抠性能余量

可选方向：

- 减少重复扫描同一段 `dataText`
- 减少正则使用，更多使用 `indexOf`
- 减少不必要的 `slice`
- 合并一些字段提取逻辑，避免重复遍历
- 继续压缩轻量元数据构建过程中的临时对象

适用时机：

- 现有方案稳定后，希望继续做细节打磨
- 希望以较低风险换取一些额外性能余量

预期收益：

- 对主线程 CPU 和短时内存分配有一定帮助
- 风险低于字节级 SSE parser

主要风险：

- 收益通常是渐进式的，不会像拆分链路那样有立竿见影的大提升
- 过度优化容易让代码可读性下降

当前建议：

- 可以作为未来的低风险优化池保留
- 当前不应优先于 Blob 主路径重构

## 推荐优先级

如果未来要继续做下一轮底层优化，建议按下面顺序考虑：

1. Blob 主路径重构
2. 字符串扫描细节优化
3. 字节级 SSE parser

原因：

- Blob 主路径对长期内存和存储收益最明确
- 字符串扫描优化投入较小、风险较低
- 字节级 SSE parser 最复杂，适合放在确有必要时再做
